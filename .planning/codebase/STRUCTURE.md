## Directory Purposes

**`backend/app/`:**
- Purpose: Core application logic; single importable package
- Contains: Market data module, route handlers, LLM integration, database init
- Key files: `main.py` (entry point), `market/__init__.py` (public API)

**`backend/app/market/`:**
- Purpose: Self-contained market data subsystem — prices, caching, streaming
- Contains: Abstract interface, two implementations (simulator + Massive), SSE streaming
- Key files: `interface.py` (contract), `cache.py` (core), `simulator.py` (default), `stream.py` (SSE)

**`backend/app/routes/`:**
- Purpose: REST API endpoint handlers grouped by feature
- Contains: Portfolio, watchlist, chat, health check routes
- Key files: `portfolio.py`, `watchlist.py`, `chat.py`, `health.py`

**`backend/app/llm/`:**
- Purpose: LLM integration — chat client, structured output parsing, mock responses
- Contains: LiteLLM wrapper, Pydantic response models, system prompts
- Key files: `client.py` (main integration), `prompts.py` (prompt templates)

**`backend/app/db/`:**
- Purpose: Database initialization, schema, seed data
- Contains: SQL table definitions, initialization logic, seed defaults
- Key files: `schema.sql` (tables), `__init__.py` (init orchestration)

**`backend/tests/`:**
- Purpose: Unit tests for backend modules
- Contains: Test modules organized to mirror `app/` structure
- Key files: `conftest.py` (fixtures), `market/test_*.py` (market data tests)

**`backend/static/`:**
- Purpose: Built static export of frontend (Next.js `output: 'export'`)
- Contains: HTML, JS, CSS, images from frontend build
- Key files: `index.html` (SPA entry point), `_next/` (bundles)

**`frontend/`:**
- Purpose: Next.js TypeScript SPA source code
- Contains: React components, pages, API client, styling
- Key files: `pages/index.tsx` (main page), `components/` (UI), `lib/api.ts` (API client)

**`test/`:**
- Purpose: E2E test infrastructure with Docker Compose
- Contains: Playwright test code, docker-compose.test.yml, config
- Key files: `docker-compose.test.yml` (test setup), `tests/` (Playwright specs)

**`db/`:**
- Purpose: Runtime Docker volume mount point for SQLite database
- Contains: `finally.db` created by backend at startup (not committed)
- Key files: `.gitkeep` (placeholder)

**`planning/`:**
- Purpose: Specification and design docs for agent consumption
- Contains: Full spec (PLAN.md), component summaries, archive
- Key files: `PLAN.md` (100% spec coverage), `MARKET_DATA_SUMMARY.md` (market data completion notes)

**`.planning/codebase/`:**
- Purpose: Generated codebase analysis documents (GSD agents)
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, CONCERNS.md
- Key files: All analysis docs (this one is STRUCTURE.md)

**`scripts/`:**
- Purpose: User-facing Docker start/stop commands
- Contains: Bash (macOS/Linux) and PowerShell (Windows) scripts
- Key files: `start_mac.sh` (primary), `start_windows.ps1` (Windows alternative)

## Naming Conventions

**Files:**
- Python modules: `snake_case.py` (e.g., `price_cache.py`)
- Test files: `test_<module>.py` (e.g., `test_cache.py`)
- TypeScript/React: `camelCase.ts` or `PascalCase.tsx` (e.g., `useApi.ts`, `PriceChart.tsx`)
- Routes: `<feature>.py` (e.g., `portfolio.py`, `watchlist.py`)

**Directories:**
- Python packages: `snake_case` (e.g., `app/market/`, `app/routes/`)
- Feature directories: `snake_case` plural (e.g., `tests/`, `components/`)
- Logical groupings: `snake_case` (e.g., `_next/`, `hooks/`)

**Classes:**
- Data models: `PascalCase` (e.g., `PriceUpdate`, `MarketDataSource`)
- Implementations: `<Name>Implementation` or `<Name>Source` (e.g., `SimulatorDataSource`, `MassiveDataSource`)

**Functions/Methods:**
- Python: `snake_case` (e.g., `create_market_data_source()`, `get_price()`)
- TypeScript: `camelCase` (e.g., `fetchPortfolio()`, `handleTradeSubmit()`)

**Variables:**
- Private (module-level): `_leading_underscore` (e.g., `_prices`, `_cache`)
- Constants: `SCREAMING_SNAKE_CASE` (e.g., `DEFAULT_DT`, `SEED_PRICES`)

## Where to Add New Code

**New Backend Feature (e.g., Analytics Dashboard):**
- Implementation: `backend/app/routes/analytics.py` — FastAPI router with endpoints
- Database queries: Add helper functions in `backend/app/db/__init__.py` or new module `backend/app/db/analytics.py`
- Tests: `backend/tests/test_analytics.py` or `backend/tests/routes/test_analytics.py`
- Schema changes: Update `backend/app/db/schema.sql` and create migration (or reinitialize for fresh installs)

**New Market Data Source (e.g., WebSocket alternative):**
- Implementation: `backend/app/market/websocket_source.py` — inherit from `MarketDataSource` abstract class
- Update: `backend/app/market/factory.py` — add new environment variable check
- Tests: `backend/tests/market/test_websocket_source.py`

**New API Endpoint (e.g., Position Details):**
- Implementation: Add to existing `backend/app/routes/<feature>.py` or new file if new feature
- Validation: Ticker normalization, business rule checks (quantity > 0, etc.)
- Response model: Define Pydantic model for response shape if complex
- Tests: Add test to corresponding test file

**Frontend Component (e.g., New Chart):**
- Implementation: `frontend/components/Chart/MyChart.tsx` — React component
- Hook (if stateful): `frontend/hooks/useChartData.ts` — Custom hook for data fetching/state
- Tests: `frontend/__tests__/components/Chart/MyChart.test.tsx` — React Testing Library
- Styling: Tailwind CSS classes inline or `frontend/styles/chart.css` if complex

**New Background Task:**
- Implementation: Async function in appropriate module, started in `main.py` lifespan context
- Example: Market data source task (already exists in `app/market/`)
- Lifecycle: `start()` called on app startup, `stop()` called on shutdown via `lifespan` context manager

**Shared Utilities:**
- Location: `backend/app/utils/` (to be created as needed) or per-module helpers
- Examples: `portfolio_valuation()` function shared by portfolio endpoint + snapshot task
- Pattern: Pure functions, no side effects, simple contracts

**Tests:**
- Unit: Co-located with feature in `backend/tests/` mirroring `app/` structure
- Integration: `backend/tests/test_integration.py` for cross-module flows (optional, keep focused on market data first)
- E2E: `test/tests/` with Playwright

## Special Directories

**`backend/static/`:**
- Purpose: Built frontend static files served by FastAPI
- Generated: Yes (during Docker build, `next build && next export`)
- Committed: No (in .gitignore, rebuilt in container)
- How to populate: Frontend build copies output here during Docker build stage

**`db/`:**
- Purpose: Docker volume mount target for SQLite persistence
- Generated: Yes (finally.db created by backend on first run)
- Committed: No (.gitkeep only; db/ itself is gitignored or excluded from bind mounts)
- How to populate: Backend initialization during lifespan startup

**`.planning/codebase/`:**
- Purpose: GSD agent-generated codebase analysis documents
- Generated: Yes (by `/gsd:map-codebase` agent command)
- Committed: Yes (treated as project documentation, checked in)
- How to populate: Run `/gsd:map-codebase` command in Claude Code

**`planning/archive/`:**
- Purpose: Historical design docs, decision records, superseded specs
- Generated: No (manually created during project development)
- Committed: Yes (preserved for reference)
- How to populate: Move superseded docs here as project evolves
